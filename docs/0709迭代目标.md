## 1.当前架构距离目标的差距

  当前系统的基础能力：

  - 已有用户体系、JWT、记录 CRUD。
  - 已支持 Markdown 编辑和上传。
  - 已有 record_chunk，保存 chunk 和 embedding。
  - 已有 QA 历史与引用表。
  - 已有 ECharts 图谱展示。
  - 已有推荐接口和局部图谱接口。
  - 已有 AI 摘要、关键词、问答接口，但都是 Mock 实现。

  主要差距：

   目标能力                                         当前状态                                                  差距
  ━━━━━━━━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  
   Raw Markdown               record.content 直接存正文                          没有 document 状态、版本、来源、处理状态
  ──────────────────────  ────────────────────────────────────────────────────────  
   Knowledge Compiler      分散在 RecordService 的 create/update/summary/keywords    没有编译任务、阶段、产物、错误重试
  ──────────────────────  ────────────────────────────────────────────────────────  
   Structured Knowledge    只有 summary、keywords 字符串                缺少概念、实体、主题、原子知识、结构化 JSON
  ──────────────────────  ────────────────────────────────────────────────────────  
   Knowledge Graph         GraphService 运行时拼标签/关键词                   没有持久化节点、边、置信度、来源
  ──────────────────────  ────────────────────────────────────────────────────────  
   Hybrid Retrieval           只遍历全部 chunk 做 cosine                                缺少 BM25/关键词检索、图谱扩展、rerank、查询改写
  ──────────────────────  ────────────────────────────────────────────────────────  
   AI Q&A                          Mock answer，简单拼 contexts                            缺少 grounded prompt、引用校验、拒答策略、追问上下文
  ──────────────────────  ────────────────────────────────────────────────────────  
   Knowledge Evolution                QA 保存历史                                               没有从问答沉淀新知识、冲突检测、健康检查

  **本质问题：**

  1. record 既是原始文档，又是知识文档，又是业务记录，职责过重。
  2. 编译逻辑被动、同步、不可观测。
  3. chunk 是检索副产品，不是知识结构的一部分。
  4. 图谱是展示层临时生成的，不是知识层真实资产。
    5. RAG 只有“向量召回”，还没有“知识库推理”。

## 2.需要重构的模块

###  2.1 RecordService 重构为 Document Service + Compiler Trigger

  当前 RecordService 同时负责：

  - 笔记 CRUD
  - 标签同步
  - chunk 重建
  - embedding 生成
  - 摘要生成
  - 关键词提取
  - 推荐
  - timeline
  - feedback

  建议拆分：

  DocumentService
  - 保存 Raw Markdown
  - 管理 document 状态
  - 管理版本
  - 触发 compiler job

  KnowledgeCompilerService
  - 解析 Markdown
  - 清洗内容
  - 智能分块
  - 调用 LLM/Embedding
  - 保存结构化知识
  - 生成图谱节点和关系

  ChunkService
  - 分块策略
  - chunk embedding
  - chunk 元数据

  KnowledgeGraphService
  - 节点/边持久化
  - 局部图谱查询
  - 图谱扩展

  RetrievalService
  - 向量召回
  - 关键词召回
  - 图谱扩展召回
  - rerank
  - context pack

  QaService
  - 问答编排
  - 引用保存
  - 问答沉淀为知识

  RecommendationService
  - 相关笔记
  - 相关概念
  - 阅读路径

  RecordService 可以短期保留为兼容层，对外仍暴露 /api/records，内部逐步迁移到 document 和 knowledge_* 表。

###  2.2 AIService 重构为多能力接口

  当前：

  generateSummary()
  extractKeywords()
  answer()
  companionFeedback()

  升级为：

  LLMService
  - summarizeDocument

  AiProvider
  - OpenAI / DeepSeek / Qwen / Ollama / Mock

  Mock 可以继续保留用于课程演示和测试，但不能作为架构中心。

###  2.3 GraphService 重构

  当前图谱节点来自：

  - record
  - tag
  - keyword
  - mood
  - type
  - recommendations

  这不是“知识图谱”，更像“笔记元数据关系图”。

  需要改为从持久化表读取：

  KnowledgeNode
  - Note
  - Concept
  - Entity
  - Question
  - Claim / AtomicKnowledge

  KnowledgeRelation
  - contains
  - related
  - derived_from
  - similar_to
  - mentions
  - answers
  - conflicts_with
  - supports

###  2.4 QaService 重构

  当前 QaService.ask() 做的是：

  question embedding
  遍历所有 chunk
  cosine score
  TopK
  AI answer
  保存 QA 和引用

  问题：

  - 全表遍历，不可扩展。
  - 只有向量召回。
  - 没有关键词召回。
  - 没有图谱扩展。
  - 没有 rerank。
  - 没有 citation grounding 检查。
  - 没有多轮追问上下文。
  - 没有把高价值问答沉淀回知识库。

  建议让 QaService 只负责问答流程编排，检索交给 RetrievalService。

## 4.Knowledge Compiler 设计

  Knowledge Compiler 是这次升级的核心。它不应该是几个按钮上的 AI 功能，而应该是后端独立流水线。

 ### 4.1 编译触发时机

  触发源：

  - 新建 Markdown。
  - 上传 Markdown。
  - 更新文档。
  - 手动重新编译。
  - 模型版本变化。
  - Health Check 发现结构过期。
  - QA 产生新知识后生成派生文档。

  触发后：

  document.status = PROCESSING
  compiler_job.status = PENDING

第一阶段可以同步执行；更合理的是异步任务：

  Controller
   ↓
  DocumentService.save()
   ↓
  CompilerJobService.enqueue()
   ↓
  KnowledgeCompilerService.run(jobId)

###  4.2 编译阶段

  流水线：

  1. Parse Markdown
  2. Clean Content
  3. Build Document Outline
  4. Semantic Chunking
  5. Generate Chunk Embeddings
  6. Generate Document Summary
  7. Extract Keywords
  8. Extract Concepts
  9. Extract Entities
  10. Generate Atomic Claims
  11. Generate Knowledge Relations
  12. Persist Structured Knowledge
  13. Build Graph
  14. Refresh Retrieval Index
  15. Mark Completed

###  4.3 Markdown 解析

  不再按固定字符长度切分。

  目前项目的 chunks() 是固定步长切分，这会破坏标题语义和段落结构。

  建议使用：

  - 标题层级：#、##、###
  - 段落
  - 列表
  - 代码块
  - 表格
  - 引用块

  输出中间结构：

  {
    "title": "RAG 系统设计",
    "sections": [
      {
        "path": ["RAG 系统设计", "Retriever"],
        "heading": "Retriever",
        "content": "...",
        "blocks": []
      }
    ]
  }

  Java 可选实现：

  - CommonMark / flexmark-java 解析 Markdown AST。
  - 简化版可以先按标题正则解析，但最终建议 AST。

  ### 4.4 智能分块策略

  分块目标不是“均匀长度”，而是“语义完整 + 可引用”。

  推荐规则：

  优先按标题章节切
  章节过长再按段落切
  段落过长再按句子切
  保留 section_path
  chunk 目标 300-800 tokens
  chunk overlap 50-100 tokens
  代码块/表格尽量不拆

  每个 chunk 需要保存：

  - 原文位置。
  - 所属标题。
  - 所属文档版本。
  - token 数。
  - embedding 模型版本。
  - content hash。

###  4.5 LLM 结构化抽取

  Compiler 应让 LLM 输出严格 JSON，不要自由文本。

  建议 Schema：

 ```
 {
    "summary": "string",
    "keywords": ["string"],
    "concepts": [
      {
        "name": "Hybrid Retrieval",
        "description": "融合向量、关键词和图谱扩展的检索方式",
        "confidence": 0.92
      }
    ],
    "entities": [
      {
        "name": "Spring Boot",
        "type": "Technology",
        "description": "后端框架",
        "confidence": 0.95
      }
    ],
    "claims": [
      {
        "text": "当前系统的 QaService 只基于 chunk embedding 进行 TopK 检索",
        "source": "chunk_id",
        "confidence": 0.9
      }
    ],
    "relations": [
      {
        "source": "Hybrid Retrieval",
        "target": "RAG",
        "type": "related",
        "evidence": "增强 RAG 架构包含向量检索、关键词检索和知识关系扩展",
        "confidence": 0.88
      }
    ]
  }
 ```



  ### 4.6 编译产物持久化

  不要只写回 record.summary 和 record.keywords。

  应保存到：

```
  document
  knowledge_chunk
  knowledge_node
  knowledge_relation
  document_knowledge_node
  keyword_index
```

  并保留编译版本：

```
  compiler_version = "kc-v1"
  embedding_model = "text-embedding-xxx"
  llm_model = "xxx"
```

  这样未来重新编译、模型升级、排查问答错误都有依据。

###  4.7 幂等与重编译

  重编译时不能简单全删全插，建议：

  

```
content_hash 相同：跳过
  chunk_hash 相同：复用 embedding
  node normalized_name 相同：合并
  relation source+target+type 相同：更新 weight/confidence
  旧节点无来源：降权或标记 stale
```



## 5.知识图谱升级方案

  当前图谱是“展示时生成”，后续升级为“知识层资产”。

###  5.1 节点模型

  建议节点类型：

  NOTE
  - 一篇 Markdown 文档或派生知识文档

  CONCEPT
  - 抽象概念，例如 RAG、Knowledge Compiler、向量检索

  ENTITY
  - 具体实体，例如 Spring Boot、H2、ECharts

  CLAIM
  - 原子知识/事实判断，例如 “当前 embedding 是 hash mock 实现”

  QUESTION
  - 用户问过且有知识价值的问题

  需求文档写的是 Note、Concept、Entity、Question。工程上建议增加 CLAIM，项目目标的LLM Knowledge Base 的关键不是只
  存文档，而是把文档编译为更小、更可组合的知识单元。

###  5.2 关系模型

  基础关系：

```
  contains       Note contains Concept/Entity/Claim
  mentions       Chunk mentions Entity
  derived_from   Node derived_from Document/Chunk/Question
  related        Concept related Concept
  similar_to     Note similar_to Note
  answers        Claim answers Question
  supports       Claim supports Claim
  conflicts_with Claim conflicts_with Claim
```

  每条边都应有：

```
  weight
  confidence
  evidence_text
  source_document_id
  source_chunk_id
```

  否则图谱会变成不可解释的装饰图。

###  5.3 图谱构建来源

  图谱边来自四类来源：

  1. Markdown 结构：
      - document contains section
      - section contains chunk

  2. LLM 抽取：
      - note contains concept
      - entity related concept
      - claim derived_from chunk

  3. Embedding 相似：
      - note similar_to note
      - concept similar_to concept
      - chunk similar_to chunk

  4. 用户行为：
      - question derived_from qa_record
      - answer references claim/chunk
      - accepted evolution creates node/relation

###  5.4 图谱查询能力

  接口建议：  GET /api/graph

  - 全局图谱，支持 type/filter/limit

  GET /api/graph/node/{id}
  - 节点详情

  GET /api/graph/node/{id}/neighbors?depth=1
  - 局部图谱

  GET /api/documents/{id}/graph
  - 文档局部图谱

  GET /api/qa/history/{id}/graph
  - 问答相关图谱

  5.5 图谱不应替代数据库

  关系型数据库仍保存事实表；图谱先用 knowledge_node 和 knowledge_relation 表即可。等数据规模和关系查询复杂度上来，再考虑
  Neo4j。当前课程项目没有必要一开始引入图数据库。

## 6.RAG 升级方案

  当前 RAG 最大问题是“只做向量 TopK”。目标是 Hybrid Retrieval：

   + ```
        Query Understanding
     ↓
     Vector Retrieval
  
      + Keyword Retrieval
     + Graph Expansion
        ↓
         Merge & Deduplicate
           ↓
         Rerank
           ↓
         Context Packing
          ↓
         Grounded Answer
          ↓
         Citation Check
          ↓
         Knowledge Evolution
  ```
  
  

###  6.1 Query Understanding

  对用户问题先做轻量分析：

```
  {
    "rewritten_query": "当前项目距离 AI 原生知识库还有哪些架构差距",
    "keywords": ["AI原生知识库", "Knowledge Compiler", "RAG", "知识图谱"],
    "concepts": ["Knowledge Compiler", "Hybrid Retrieval"],
    "entities": ["CoBloom"],
    "intent": "analysis",
    "need_graph_expansion": true
  }
```

  可以由 LLM 输出，也可以先用规则 + 分词实现。

  6.2 Vector Retrieval

  候选源：

```
  knowledge_chunk.embedding
  knowledge_node.embedding
  document.embedding 或 summary.embedding
```

  召回：

```
  TopK chunks
  TopK concepts
  TopK claims
```

  短期仍可用 Java 内存 cosine；正式建议 PostgreSQL + pgvector。

 ### 6.3 Keyword Retrieval

  不只依赖 embedding。很多项目名、类名、接口名、技术名适合关键词检索。

  实现路径：

  **P0 简化版：**

  keyword_index 表
  term like / exact match
  标题、section_path、keywords 加权

  **正式版：**

  PostgreSQL tsvector + GIN
  中文分词可用 pg_jieba 或应用层分词

  权重示例：

  title match: 3.0
  heading match: 2.0
  keyword match: 2.0
  content match: 1.0

  ### 6.4 Graph Expansion

  从初始召回结果扩展：

```
  chunk -> source document
  document -> concepts/entities/claims
  concept -> related concepts
  question -> answered_by claims
  similar document -> useful chunks
```

  限制：

  depth <= 2
  每个节点最多扩展 N 条边
  relation confidence >= threshold

  图谱扩展的作用不是盲目加上下文，而是补充“邻近知识”。

###  6.5 Candidate Merge

  所有候选统一成：

```
  {
    "type": "chunk|node|claim|document",
    "id": 123,
    "text": "...",
    "source_document_id": 1,
    "source_chunk_id": 10,
    "scores": {
      "vector": 0.81,
      "keyword": 0.6,
      "graph": 0.4,
      "recency": 0.2
    },
    "final_score": 0.73
  }
```

  初始打分：

  + ```
      final_score =vector_score * 0.45
    
      + keyword_score * 0.25
      + graph_score * 0.20
      + recency_score * 0.05
      + authority_score * 0.05
    ```

    

###  6.6 Rerank

  P0 可以用加权分数。

  P1 引入 LLM rerank 或 cross-encoder rerank：

  输入：query + candidate snippets
  输出：相关性 0-1、是否可回答、原因

###  6.7 Context Packing

  不用简单 String.join(contexts)。

  应构造有来源的上下文：

```
  [Source 1]
  document: xxx
  section: xxx
  chunk_id: 12
  content: ...

  [Source 2]
  concept: Knowledge Compiler
  description: ...
  derived_from: document 3 / chunk 8
```

  控制规则：

  优先 claims 和高分 chunks
  同一文档避免重复塞太多 chunk
  保留标题和 section_path
  总 token 控制在模型窗口内

 ### 6.8 Grounded Answer

  Prompt 原则：

  只能基于提供的上下文回答
  每个关键结论必须引用 Source ID
  信息不足必须明确说不足
  不要把模型常识混入用户知识库

  返回结构建议：

```
  {
    "answer": "string",
    "citations": [
      {
        "source_id": "S1",
        "document_id": 1,
        "chunk_id": 12,
        "quote": "短引用"
      }
    ],
    "confidence": 0.82,
    "missing_information": ["缺少真实 LLM Provider 配置"]
  }
```

 ### 6.9 Citation Check

  生成后再做一次校验：

  answer claim 是否能被 citations 支持
  citation 是否真实来自 chunk
  没有引用的断言降级或移除

  计划先做规则校验：

  - answer 中必须存在引用。
  - 引用 chunk 必须在本次 retrieval candidates 内。
  - 无候选时必须拒答。

###  6.10 Knowledge Evolution

  QA 不应只保存历史。高价值问答可以成为新知识。

  流程：

```
  用户提问
   ↓
  RAG 回答
   ↓
  系统生成 evolution suggestion
   ↓
  用户接受/拒绝
   ↓
  创建 Question 节点
   ↓
  创建 Claim 或 Note
   ↓
  建立 derived_from / answers / references 关系
   ↓
  触发局部重编译
```

  示例：

  Question: 当前项目如何升级为 AI 原生知识库？
  Claim: 当前项目缺少持久化知识节点与关系。
  Relation: Claim answers Question
  Relation: Claim derived_from QaRecord + cited chunks

  这就是“AI Q&A → Knowledge Evolution”。

## 7.推荐落地路线

###  P0：先把流水线立起来

  目标：从“保存笔记即切 chunk”升级为“保存文档后编译”。

  要做：

  - 新增 document.status 或在 record 增加 status。
  - 新增 compiler_job。
  - 抽出 KnowledgeCompilerService。
  - Markdown 按标题/段落切 chunk。
  - 自动生成 summary、keywords。
  - 保存 concepts/entities 到 knowledge_node。
  - 保存 contains/derived_from 到 knowledge_relation。
  - QA 改为走 RetrievalService。
  - 图谱改为读取 knowledge_node/relation。

###  P1：升级图谱和 Hybrid Retrieval

  要做：

  - 引入 Concept、Entity、Claim。
  - 建立图谱扩展召回。
  - 加关键词索引。
  - 检索候选融合打分。
  - QA 返回结构化 citations。
  - 记录 retrieval trace。
  - 推荐改成 vector + keyword + graph + tag。

###  P2：知识演化和健康检查

  要做：

  - QA 可沉淀为 Question/Claim。
  - Health Check 检查重复、冲突、缺失关系。
  - 文档版本管理。
  - 重编译复用 embedding。
  - PostgreSQL + pgvector 替换 H2/JSON embedding。
  - 接入真实 LLM 和 Embedding API。
