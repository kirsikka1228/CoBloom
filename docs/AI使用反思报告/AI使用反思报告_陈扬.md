# AI 使用反思报告

## 一、我的分工

我在 CoBloom 项目中主要负责前端开发，包括登录页面、笔记列表与详情、笔记编辑、问答页面、时间线、知识图谱展示，以及前端和后端接口的对接。

项目使用 Vue 3、Vue Router、Pinia、Axios、Element Plus、ECharts 和 Marked。除了完成页面功能，我还负责统一页面风格、处理登录状态、路由跳转、接口报错和加载状态。

## 二、我在哪些地方使用了 AI

这次开发中，我主要把 AI 当成辅助工具，而不是让它直接替我完成整个前端。

### 1. 页面布局和视觉风格参考

项目早期页面的功能比较分散，登录页、笔记页和问答页的风格也不统一。我使用 AI 帮助分析页面层级，讨论过侧边导航、主内容区、卡片和按钮应该怎样组织，也让它提供过纸张、拼贴和复古风格的实现建议。

不过 AI 给出的设计有时会堆太多渐变、阴影和装饰，实际放进页面后比较乱。我没有直接全部采用，而是保留了纸张、胶带和手绘按钮这些与项目主题比较合适的元素，删掉了一些影响阅读的效果。

### 2. Vue 和 CSS 问题排查

我使用 AI 辅助排查过一些前端问题，例如：

- Vue Router 刷新后页面路径找不到；
- 登录成功后怎样回到用户原来访问的页面；
- Token 失效后怎样清理登录状态；
- ECharts 页面切换后怎样释放图表实例；
- 弹窗和背景图片在不同屏幕比例下怎样定位；
- CSS 层级、定位和响应式布局问题。

AI 通常能够给出几种可能原因，但真正定位时仍需要我查看浏览器控制台、Network 请求、Vue 状态和实际 DOM。部分问题并不是 AI 第一次判断的原因，所以我会逐项验证，而不是看到一个答案就直接修改。

### 3. 接口调用模板和重复代码

对于加载状态、错误提示、删除确认、Axios 请求等重复结构，我参考过 AI 给出的代码写法。例如请求前设置 `loading`，结束后在 `finally` 中恢复；删除数据前使用确认弹窗；通过统一的 Axios 实例处理后端错误。

这些代码最后都结合项目接口进行了修改。我需要确认请求路径、请求方式、字段名称和后端返回结构是否一致。例如问答来源在前端需要展示标题、片段和相似度，如果只照着通用示例写，很容易出现字段对不上但页面不报明显错误的情况。

### 4. 代码理解和验收准备

在验收准备阶段，我使用 AI 帮助梳理了前端目录结构和页面调用关系，也让它模拟老师可能提出的问题。这个过程对我比较有帮助，因为它让我发现自己虽然能写出页面，但一开始不一定能完整说明为什么需要 Pinia、为什么要使用请求拦截器、路由守卫解决了什么问题。

整理之后，我能够用自己的语言说明一条请求从页面发出，到 `http.js` 添加 Token，再到后端返回数据，最后由 Vue 响应式状态更新页面的过程。

## 三、我没有让 AI 做什么

本次验收规定的 AI 禁飞区包括知识图谱实体关系抽取、RAG 检索相关性排序和问答答案来源溯源。这三部分的核心实现位于后端，不属于我的主要分工。我没有使用 AI 替团队编写或修改这些禁飞区的核心算法。

在前端，我只负责展示后端返回的图谱、问答结果和引用来源。例如 `Graph.vue` 使用 ECharts 显示节点和边，`QaAsk.vue`、`QaDetail.vue` 展示回答引用的笔记标题、片段和相关性分数。前端没有决定哪些实体之间存在关系，也没有计算 RAG 最终排名。

对于自己负责的前端代码，我没有直接提交一整份不了解的 AI 输出。即使使用了 AI 建议，我也会先确认：

1. 代码对应哪个页面需求；
2. 调用了哪个后端接口；
3. 成功和失败时页面分别怎样变化；
4. 页面离开时是否需要清理监听器或图表实例；
5. 是否会影响登录状态和其他页面。

如果一段代码我还不能解释，我不会把“能运行”当成已经完成。

## 四、AI 带来的帮助

AI 对我帮助最大的是缩短查资料和尝试方案的时间。以前遇到一个 CSS 定位或 Vue 生命周期问题，我可能需要在很多网页之间反复搜索；使用 AI 后，可以先得到一个排查方向，再结合浏览器工具验证。

它还帮助我补充了一些容易忽略的边界情况，例如：

- 请求过程中禁用按钮，避免重复提交；
- 使用 `finally` 保证加载状态能够恢复；
- 删除操作增加二次确认；
- ECharts 监听窗口缩放，并在组件销毁时 `dispose`；
- 登录失效后保存原路径，重新登录后再跳回来；
- 空列表、加载中和接口错误需要有不同页面状态。

这些细节不一定属于复杂算法，但会直接影响项目演示是否稳定。

## 五、AI 使用中出现的问题

### 1. AI 容易给出“看起来完整”的代码

AI 生成的组件经常包含比较完整的模板、脚本和样式，第一眼看起来可以直接使用。但接入项目后可能出现接口字段不一致、路由地址不一致、重复样式或与现有状态管理冲突的问题。

我后来形成的习惯是先拆开检查，而不是整段复制：先确认数据结构，再接请求，最后调整页面和样式。

### 2. 使用 AI 写代码容易造成理解不够清晰

使用 AI 生成代码虽然速度比较快，但如果没有逐行检查，很容易只知道这段代码可以运行，却不知道每个变量、方法和判断具体解决了什么问题。代码较短时可能暂时看不出影响，等到接口字段变化、页面状态不正确或者不同组件互相影响时，就很难判断问题究竟出在哪里。

我在调试时发现，如果代码不是自己一步步写出来并验证的，定位问题会比较慢，甚至可能为了修一个小问题反复让 AI 修改，最后代码越来越复杂。因此使用 AI 给出的代码后，我需要重新梳理执行流程，确认请求从哪里发出、状态在哪里修改、失败时会进入哪个分支，并通过控制台和 Network 面板验证。只有自己能够解释和调试的代码，才算真正完成。

### 3. 视觉建议不一定适合真实页面

AI 很容易给出效果明显的设计，但不一定考虑中文长度、不同分辨率、表单可用性和内容变多后的情况。登录封面在大屏幕上合适，不代表在较窄窗口中也不会遮挡。

所以视觉部分必须实际打开页面测试，而不是只看代码或静态截图。

### 4. AI 可能让人跳过理解过程

如果遇到问题就直接让 AI 改，短期会比较快，但之后老师问“为什么这样写”时很难回答。比如只记住路由守卫能跳转是不够的，还需要理解 `requiresAuth`、Token、`fetchMe` 和重定向参数之间的关系。

这次准备验收时，我明显感觉到“做出来”和“讲清楚”是两件不同的事。

## 六、我怎样检查 AI 建议

我主要通过下面几种方式验证：

- 使用浏览器控制台检查 JavaScript 报错；
- 使用 Network 面板检查请求 URL、状态码和返回内容；
- 刷新页面，确认登录状态能否恢复；
- 使用错误密码和空表单测试异常情况；
- 在不同页面之间切换，检查路由和菜单状态；
- 新建、修改、删除笔记，确认页面和后端数据一致；
- 分别测试空列表、加载中、接口失败和正常数据；
- 构建前端，确认 `npm run build` 能够通过；
- 对照后端 Controller 和 DTO 检查接口字段，而不是只参考 AI 的描述。

如果 AI 的建议不能通过这些检查，就需要继续定位原因或者放弃该方案。

## 七、这次开发后的认识

我认为 AI 更适合做“提供候选方案”和“帮助排查”，不适合替代开发者做最终判断。前端代码看起来没有复杂算法，但它涉及接口、状态、路由、生命周期、样式和用户操作，任何一部分没有理解都可能在演示时出问题。

这次项目让我意识到，正确使用 AI 的关键不是少用或多用，而是知道哪些内容可以交给它辅助、哪些决定必须由自己负责。对于重复代码和资料查询，可以利用 AI 提高效率；对于项目结构、核心逻辑、数据安全和验收禁飞区，必须自己理解并验证。

如果重新做一次，我会更早统一接口字段和页面组件结构，先完成基础功能与异常状态，再集中处理视觉效果。同时会补充前端测试和接口 Mock，减少每次都依赖完整后端环境进行验证。

## 八、总结

在 CoBloom 前端开发中，我使用 AI 辅助了布局讨论、CSS 调试、Vue 问题排查、重复请求代码和验收复习，但最终页面结构、接口对接、功能验证和代码取舍由我自己完成。对于知识图谱实体关系抽取、RAG 相关性排序和答案来源溯源三个禁飞区，我只负责前端展示，没有参与用 AI 生成其核心后端实现。

我认为这次 AI 使用总体上提高了开发效率，也暴露出自己在代码理解和测试意识上的不足。之后我会继续使用 AI，但会把“能够解释、能够验证、能够承担结果”作为是否采用建议的标准。
