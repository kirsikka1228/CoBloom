# Sprint 2 安全加固挑战报告

项目名称：AI 知识库与智能问答（CoBloom）  
检查日期：2026-07-14  
检查依据：[OWASP Top 10:2025](https://owasp.org/Top10/)

## 一、检查结论

本次不是对 OWASP Top 10 的全部风险做“已经绝对安全”的结论，而是结合当前代码完成一次针对性自查。最终修复了四类可以复现的问题：

1. 真实 LLM API Key 和固定 JWT Secret 写入版本库；
2. CORS 允许任意来源、任意请求头和任意方法；
3. H2 Console、Swagger 对未认证用户开放，并关闭 Frame 防护；
4. Markdown 经过 `marked` 后直接交给 `v-html`，存在存储型 XSS 风险。

对应 OWASP Top 10:2025 中的 A01 Broken Access Control、A02 Security Misconfiguration、A04 Cryptographic Failures 和 A05 Injection。

## 二、修复一：密钥不再写入仓库

### 修改前

`application.yml` 中直接保存固定 JWT Secret 和真实 LLM API Key。任何能读取仓库、构建产物或配置文件的人都可能取得这些凭据。

```yaml
cobloom:
  jwt-secret: "<仓库中的固定密钥>"
  llm:
    api-key: "sk-***"
```

风险分类：A04:2025 Cryptographic Failures、A02:2025 Security Misconfiguration。

### 修改后

```yaml
cobloom:
  jwt-secret: ${COBLOOM_JWT_SECRET}
  llm:
    api-key: ${COBLOOM_LLM_API_KEY:}
```

同时完成：

- 新增 `.env.example`，只提供占位符；
- `.env` 继续由 `.gitignore` 排除；
- Docker Compose 从运行环境读取密钥；
- README 和部署文档补充环境变量设置方法；
- 测试使用明确标注的 test-only JWT Secret，不依赖生产密钥；
- 新增测试，防止 `application.yml` 再次出现 `api-key: sk-` 或演示 JWT Secret。

Windows PowerShell 5.1 可使用以下方式生成 JWT Secret：

```powershell
$jwtBytes = New-Object byte[] 48
$jwtRng = [System.Security.Cryptography.RandomNumberGenerator]::Create()
$jwtRng.GetBytes($jwtBytes)
$env:COBLOOM_JWT_SECRET = [Convert]::ToBase64String($jwtBytes)
$jwtRng.Dispose()
```

### 仍需人工完成

真实 LLM API Key 曾经进入 Git 记录。删除当前文件中的 Key 不能让历史提交中的 Key 自动失效，必须登录服务商后台撤销旧 Key，再创建新 Key 并放入本机环境变量。这个步骤完成前，密钥泄露风险不能算完全关闭。

## 三、修复二：限制 CORS

### 修改前

```java
config.addAllowedOriginPattern("*");
config.addAllowedHeader("*");
config.addAllowedMethod("*");
```

任何网站都可以尝试从浏览器向接口发起跨域请求。OWASP Top 10:2025 在 A01 Broken Access Control 中明确将 CORS 配置错误列为访问控制风险。

### 修改后

```java
config.setAllowedOrigins(allowedOrigins);
config.setAllowedHeaders(List.of("Authorization", "Content-Type", "Accept"));
config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
```

允许来源通过 `COBLOOM_CORS_ALLOWED_ORIGINS` 配置，默认仅为：

```text
http://localhost:5173
http://localhost
```

程序会拒绝空配置和 `*`，避免部署时误操作重新放开全部来源。

### 验证结果

| 请求 | 结果 |
|---|---|
| Origin 为 `http://localhost:5173` 的预检请求 | HTTP 200，返回对应 Allow-Origin |
| Origin 为 `https://evil.example` 的预检请求 | HTTP 403，不返回 Allow-Origin |

## 四、修复三：关闭公开调试入口并恢复安全响应头

### 修改前

```java
.requestMatchers("/h2-console/**", "/swagger-ui/**", "/v3/api-docs/**").permitAll()
.headers(headers -> headers.frameOptions(frame -> frame.disable()))
```

并且：

```yaml
spring:
  h2:
    console:
      enabled: true
```

这会把数据库调试入口和接口说明暴露给未登录用户，同时移除点击劫持防护。风险分类为 A01:2025 Broken Access Control 和 A02:2025 Security Misconfiguration。

### 修改后

- H2 Console 默认 `enabled: false`；
- 删除 H2、Swagger 和 OpenAPI 的匿名放行规则；
- 删除 Frame Options 禁用代码，恢复 Spring Security 默认的 `X-Frame-Options: DENY`；
- README 和部署文档不再把 H2、Swagger 描述成默认公开入口。

### 验证结果

| 请求 | 状态码 | 安全响应头 |
|---|---:|---|
| `GET /h2-console`，未登录 | 403 | `X-Frame-Options: DENY` |
| `GET /swagger-ui/index.html`，未登录 | 403 | `X-Frame-Options: DENY` |

## 五、修复四：清理 Markdown，阻止存储型 XSS

### 修改前

两个页面把用户可控 Markdown 直接转换为 HTML：

```js
const html = computed(() => marked.parse(record.value?.content || ''))
```

随后通过 `v-html` 插入页面。`marked` 负责 Markdown 解析，但不负责把不可信 HTML 清理为安全内容。恶意笔记可以包含事件属性、危险链接或嵌入标签。

风险分类：A05:2025 Injection。

### 修改后

新增统一方法：

```js
export function renderSafeMarkdown(source) {
  const rendered = marked.parse(String(source || ''))
  return DOMPurify.sanitize(rendered, SANITIZE_OPTIONS)
}
```

并完成：

- `RecordDetail.vue` 和 `RecordEdit.vue` 统一调用 `renderSafeMarkdown()`；
- 禁止 `iframe`、`object`、`embed`、`form`、`input`、`style` 等不需要的标签；
- 禁止内联 `style` 和 `data-*` 属性；
- DOMPurify 继续清理事件处理属性和危险 URL；
- `npm audit` 结果为 0 vulnerabilities；
- 前端生产构建成功。

建议现场使用以下内容演示，修复后文字可以显示，但危险内容不能执行：

```markdown
# XSS Test
<img src=x onerror="alert('xss')">
[danger](javascript:alert('xss'))
<iframe src="https://evil.example"></iframe>
```

## 九、结论

本次完成的修改不是只隐藏页面入口，而是在配置、访问控制、响应头和 HTML 渲染四个层面进行加固。自动化测试和真实 HTTP 请求均通过，原有后端覆盖率门禁和前端构建没有被破坏。

仍需在提交前完成两件事：撤销曾经提交过的旧 LLM API Key；提交时排除 `backend/target` 等生成文件。安全加固是持续过程，本报告不能替代后续的依赖扫描、权限测试和日志审计。
