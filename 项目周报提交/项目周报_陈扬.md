**四、遇到的问题与困难**

* **技术问题：**
    * 登录状态需要在页面刷新后恢复，同时还要处理 Token 失效、受保护路由跳转和退出登录等情况。
    * **已尝试的解决方案：** 使用 Pinia 保存用户状态，将 Token 持久化到 `localStorage`；通过 Axios 拦截器统一添加 Token；进入页面时调用 `/auth/me` 恢复用户；Token 无效时发送全局事件并跳转登录页。
    * ECharts 图谱在页面切换和窗口尺寸变化后可能出现尺寸不正确或实例未释放的问题。
    * **已尝试的解决方案：** 在组件挂载后初始化图表，监听窗口 `resize`，并在组件卸载时移除监听器和调用 `dispose()`。
    * Markdown 内容需要同时支持编辑预览和详情展示。
    * **已尝试的解决方案：** 使用 Marked 将 Markdown 转换为 HTML，通过 Vue 计算属性实时刷新预览，并统一 Markdown 展示样式。
    * 部分前端请求失败时只能看到通用错误，排查接口问题不够直观。
    * **已尝试的解决方案：** 使用浏览器 Network 面板检查请求地址、状态码和响应内容，并在 Axios 响应拦截器中优先展示后端返回的 `message`。
    * **需要帮助或待解决：** 需要后端成员进一步统一接口字段、错误状态码和异常返回格式，以减少前端适配成本。

* **非技术问题：**
    * 初始阶段页面较多，部分接口和字段仍在调整，前端容易出现已经完成的页面需要再次适配的情况。
    * **已采取的措施：** 先确定路由、Axios 和登录状态等公共框架，再逐个接入页面；与后端成员及时确认接口路径和返回结构，尽量把公共修改集中在 `http.js` 和状态管理中，避免每个页面重复修改。

* **对项目或开发流程的思考：**
    * 前端初始框架是否清晰，会直接影响后续页面开发速度。如果每个页面分别处理 Token、错误和接口地址，代码会很快重复，因此公共能力应该尽早抽到状态管理、路由和请求封装中。
    * 前后端联调不能只确认接口“能请求成功”，还需要提前统一字段名称、错误格式和空数据情况，否则页面在正常数据下可用，但遇到异常时容易出现问题。
    * 页面视觉统一应该建立在功能稳定的基础上。先确定布局和组件结构，再集中调整素材与样式，比开发一个页面就重新设计一次更有效率。

* **自我改进方面：**
    * 本周更关注功能是否能够运行，对表单边界、异常状态和测试覆盖考虑得还不够。下周会在开发每个功能时同时检查正常、空数据、加载、失败和重复操作等情况。
    * 使用 AI 或参考代码时，需要先理解执行流程再采用，避免只知道代码能够运行，却在出现问题时无法快速调试。
    * 后续会减少无意义的重复样式和未使用图片，保持前端目录清晰，并养成按功能提交 Git 记录的习惯。
