# CoBloom 核心架构决策记录 (Decisions)

本文件记录了项目开发过程中关于技术选型、业务逻辑以及核心演进的关键决策（Architecture Decision Records）。

## [ADR-001] 采用双轨分析模式（存储与AI反馈同步处理）
- **状态**: 已采纳 (Accepted)
- **上下文**: 用户在记录个人生活或情绪时，系统需要同时满足“高即时性的情感回应”和“面向未来的结构化检索”两种完全不同的业务诉求。
- **决策**:
  - 采用双轨处理模式：用户提交一条 `GrowthRecord` 后，后端同时触发两个链路。
  - **情感陪伴轨**：立即或异步调用大模型，生成温暖的陪伴话术 (`CompanionFeedback`)，并提取当前情绪标签，让用户在刷新页面后能立刻感受到“记录被看见、成长被回应”。
  - **知识图谱/RAG轨**：将文本切片为 `RecordChunk`，同步构建图谱连接，为长期的智能问答做底层底座支撑。
- **后果**: 保证了产品在情感层面的温度（响应迅速且共情），同时兼顾了文本持久化与切片的准确性。

## [ADR-002] 采用自定义内嵌文本相似度模块（TextSimilarity）替代独立向量数据库
- **状态**: 已采纳 (Accepted)
- **上下文**: 在项目初期与本地开发阶段，引入商用或独立的向量数据库（如 Milvus, Pinecone）会极大地增加本地部署环境的复杂度，且对于个人私密知识库而言，单个用户的数据量在短期内不会达到百万级。
- **决策**: 
  - 本阶段暂不引入复杂的外部向量数据库拓扑，直接在后端服务中封装 `EmbeddingService` 和 `TextSimilarity` 逻辑。
  - 文本向量化后，将向量或核心特征随 `RecordChunk` 直接存在轻量级数据库中，利用 Java 内存计算或 SQL 进行快速的相似度余弦计算排序。
- **后果**: 极大地简化了系统的部署难度和运行成本。但在未来如果用户量暴增或单用户记录突破数万条时，需要重构 `EmbeddingService` 以适配专业的向量中间件。

## [ADR-003] 基于 JWT 的无状态认证与 UserContext 本地线程绑定
- **状态**: 已采纳 (Accepted)
- **上下文**: 系统包含大量高度敏感的个人私密情绪和成长隐私，任何接口的越权（如 A 用户看到了 B 用户的图谱或 AI 问答）都是不可接受的。
- **决策**:
  - 前后端交互完全基于 JWT。后端配置 `JwtAuthFilter` 对每一个带有 Token 的请求进行严格解析。
  - 解析成功后，将当前用户的唯一凭证绑定到 `UserContext`（基于 ThreadLocal）。所有 Service 层（如 `RecordService`, `QaService`）在查询数据时，强制追加 `where user_id = UserContext.getCurrentUserId()` 过滤条件。
- **后果**: 从底层框架级别杜绝了越权漏洞（IDOR），保证了数据读写绝对属于当前登录用户。
